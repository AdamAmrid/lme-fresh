## Section 1 — Seed Script (`seed_analytics.py`)

### 1.1 Database Setup

The script connects to `sqlite:///./lme.db` using SQLAlchemy with `check_same_thread=False` (write mode, not read-only). It calls `Base.metadata.create_all(bind=engine)` unconditionally at module load time to ensure tables exist before seeding.

### 1.4 Exact Values Per Student Type

All values below are determined at the **per-session** level (reset on each of the 5 sessions).

| Parameter | `engaged` | `mixed` | `at-risk` |
|-----------|-----------|---------|-----------|
| Initial `current_score` | `uniform(0.75, 0.85)` | `uniform(0.45, 0.60)` | `uniform(0.10, 0.25)` |
| `max_hints` (session cap) | `randint(0, 1)` | `randint(1, 3)` | `randint(3, 6)` |
| `target_si` per question | `uniform(0.05, 0.20)` | `uniform(0.20, 0.55)` | `uniform(0.55, 0.90)` |
| P(is_correct) | 0.85 | 0.60 | 0.25 |
| `attempt_count` (if not unengaged) | `1` if correct, `randint(1,2)` if wrong | `randint(1, 3)` | `randint(2, 5)` |
| P(Unengaged override) | — | — | 0.35 |
| `idle_time` when Unengaged | — | — | `uniform(65, 110)` |
| `attempt_count` when Unengaged | — | — | `0` (forced) |

**Hint allocation:** A question receives +1 hint if: `total_hints < max_hints AND NOT is_correct AND random() < 0.6`. Hints are never assigned to correctly-answered questions.

---

### 1.5 Intra-Session Score Progression

`current_score` is initialized once at the start of each session (see table above). It is updated **per question**:

```python
if is_correct:
    current_score = min(1.0, current_score + random.uniform(0.02, 0.08))
```

Incorrect answers do **not** decrease `current_score`. The score is monotonically non-decreasing within a session. There is **no inter-session carry-over** — `current_score` is re-sampled at the start of every session.

---

### 1.6 Idle Time Back-Calculation

`idle_time` is not sampled randomly for non-unengaged rows. It is **back-calculated** to approximately match `target_si`:

```python
# target_si = 0.4*(idle/120) + 0.4*(attempt/5) - 0.2*(score)
required_si_component = target_si + (0.2 * current_score)
required_si_component = max(0, required_si_component)

attempt_comp = 0.4 * (attempt_count / 5.0)
idle_comp = required_si_component - attempt_comp
if idle_comp < 0: idle_comp = 0

idle_time = (idle_comp / 0.4) * 120.0
idle_time = max(1.0, min(120.0, idle_time))  # clamped to [1, 120]
```

This ensures the resulting `struggle_index` (computed from exact formulas below) approximately matches the `target_si` without being exactly equal due to clamping.

---

### 1.7 Exact Formulas for `learner_state` and `struggle_index`

**`calculate_state(is_correct, attempt_count, idle_time)` — priority order:**

```python
def calculate_state(is_correct, attempt_count, idle_time):
    if is_correct:
        return "Engaged"
    elif attempt_count == 0 and idle_time < 20:
        return "Engaged"
    elif idle_time > 60:
        return "Unengaged"
    elif idle_time >= 30 or attempt_count >= 3:
        return "Struggling"
    else:
        return "Engaged"
```

**`calculate_si(idle_time, attempt_count, current_score):`**

```python
def calculate_si(idle_time, attempt_count, current_score):
    return (0.4 * (idle_time / 120.0)) + (0.4 * (attempt_count / 5.0)) - (0.2 * (current_score / 1.0))
```

The SI formula has **no floor clamp** — it can theoretically go negative if `current_score` is high and `idle_time`/`attempt_count` are low. In practice, the clamping of `idle_time` to a minimum of 1.0 and the score cap at 1.0 constrains the theoretical minimum to approximately `(0.4 * 1/120) + 0 - (0.2 * 1) ≈ -0.197`.

---

### 1.8 Timestamp Structure

- **Across sessions:** Session `i` starts `30 - (i * 6)` days in the past, giving: Day -30, -24, -18, -12, -6 relative to script runtime. A random `timedelta(minutes=randint(0, 720))` is added to the session start as jitter (up to 12 hours within that day).
- **Within a session:** Each subsequent question row advances `current_time` by:
  ```python
  current_time = current_time + timedelta(seconds=int(idle_time) + random.randint(5, 20))
  ```
  i.e., approximately `idle_time` seconds plus 5–20 seconds of engagement time.

---

### 1.9 Database Insertion — Duplicate Handling

**Users:** Checked with `session.query(User).filter(User.email == email).first()`. If a user with that email already exists, they are **skipped** (not re-inserted). Their existing `User` object is reused for telemetry generation.

**Telemetry:** There is **no duplicate check** for `TelemetryLog` rows. Every run of `seed()` inserts `10 students × 5 sessions × 8 questions = 400 new rows unconditionally` on top of any existing data. The database is **not cleared** before seeding.

> ⚠️ The `/seed` endpoint in `analytics.py` **deletes the entire `lme.db` file** before calling the seed script (`os.remove(db_path)`). This is the only way the seed is run idempotently — not through the script itself.

## Section 2 — ML Risk Model (`models/risk_model.py`)

### 2.1 Model Class and Parameters

```python
from sklearn.linear_model import LogisticRegression
model = LogisticRegression(max_iter=500)
```

**No `StandardScaler` or any other preprocessing is applied.** Features are passed raw to `model.fit()` and `model.predict_proba()`. This is a **significant architectural deviation** — see Section 4.

---

### 2.2 Global State

```python
model = None          # LogisticRegression instance, or None in fallback
is_fallback = True    # True on startup; set to False only if ML training succeeds
training_data_cache = None  # pd.DataFrame indexed by student_id
```

---

### 2.3 Feature Computation

Session boundaries in `risk_model.py` use **date-based grouping only** (`df['date'] = pd.to_datetime(df['timestamp']).dt.date`), then `groupby('date')`. This deviates from the 3-rule boundary system used in `analytics.py`.

| Feature | Column Name in Cache | Exact Computation |
|---------|---------------------|-------------------|
| Average Struggle Index | `avg_SI` | `student_df['struggle_index'].mean()` |
| Hint Dependency | `hint_dependency_score` | `session_maxes['total_hints_requested'].sum() / max(session_maxes['answered_count'].sum(), 1)` |
| Unengaged Ratio | `unengaged_ratio` | `(learner_state == 'Unengaged').sum() / len(student_df)` |
| Average Attempts | `avg_attempt_count` | `student_df['attempt_count'].mean()` |
| Session Count | `session_count` | `student_df['date'].nunique()` |
| SI Worsening | `si_worsening` | `session_avg_si.iloc[-1] > session_avg_si.iloc[0]` (bool; `False` if < 2 sessions) |

**Note on `hint_dependency_score`:** This is computed as `total_hints_across_all_sessions / total_answered_across_all_sessions` — a global ratio. This is different from the per-session average used in the analytics router (see Section 4 — Discrepancy #3).

---

### 2.4 Training Labels

```python
label = 1 if (unengaged_ratio > 0.4 or avg_SI > 0.7) else 0
```

- `1` = High Risk
- `0` = Not High Risk
- No "medium" label — training is binary classification.

---

### 2.5 Training Conditions and Fallback

Training proceeds through a sequence of guards:

1. DB must be readable and `telemetry_logs` must be non-empty → else fallback
2. `struggle_index` column must exist → else fallback (schema mismatch)
3. Only students with `session_count >= 2` are included in training
4. `len(train_df) >= 10` → else fallback (not enough labeled rows)
5. `len(train_df['label'].unique()) > 1` → both label classes must be present → else fallback (single-class guard)

**Fallback formula (used when `is_fallback=True` or `model is None`):**
```python
risk_score = (0.4 * s_feat['avg_SI']) + (0.4 * s_feat['hint_dependency_score']) + (0.2 * s_feat['unengaged_ratio'])
risk_score = max(0.0, min(1.0, risk_score))
```

**ML path (when model is trained):**
```python
X = pd.DataFrame([res["features"]])
probs = model.predict_proba(X)[0]
risk_score = float(probs[1])  # probability of label=1 (High Risk)
```

---

### 2.6 Server Startup Behavior

`train_risk_model()` is **not called at server startup from `main.py`**. It is called at the beginning of every analytics endpoint request (`/student/{id}`, `/cohort`, `/risk`) via:
```python
train_risk_model()  # Always refresh cache before predicting
```

This means the model is retrained on every request — there is no background startup training or caching across requests.

---

### 2.7 `predict(student_id: str)` Return Value

| Field | Type | Computation |
|-------|------|-------------|
| `risk_score` | `float` | Fallback formula or `model.predict_proba(X)[0][1]` |
| `risk_label` | `str` | `"low"` if score < 0.35, `"medium"` if < 0.65, `"high"` otherwise |
| `dominant_weakness` | `str` | Rule-based logic below; default `"cognitive"` |
| `suggested_intervention` | `str` | Mapped from `dominant_weakness` |
| `features` | `dict` | Raw feature values from cache |

If `student_id` is not in `training_data_cache`, all fields return default values (`risk_score=0.0`, `risk_label="low"`, `dominant_weakness="cognitive"`) and `features={}`.

### 2.8 Dominant Weakness Logic in `risk_model.py`

```python
is_behav = (s_feat['avg_attempt_count'] > 3) or (s_feat['unengaged_ratio'] > 0.3)
is_cog   = (s_feat['hint_dependency_score'] > 0.5) or (s_feat['avg_SI'] > 0.5)
is_emot  = (s_feat['unengaged_ratio'] > 0.4) or bool(s_feat['si_worsening'])

if is_behav:   → "behavioral"
elif is_cog:   → "cognitive"
elif is_emot:  → "emotional"
# else: default stays "cognitive" (set in res initialization)
```

**This differs from the router's logic** — the router has an additional `avg_idle > 120` behavioral trigger that `risk_model.py` does not have.

| `"behavioral"` | `"Suggest a break and reset task pacing"` |
| `"cognitive"` | `"Add scaffolded hints and reduce question difficulty"` |

## Section 3 — Analytics Router (`routers/analytics.py`)

```python

```

The router opens `lme.db` in **read-only mode**. All four endpoints use this shared engine. The `/seed` endpoint, however, calls a subprocess that runs `seed_analytics.py`, which opens the DB in **write mode** via its own engine instance.

### 3.3 Session Boundary Detection (Router)

Used identically in both `/student/{id}` and `/cohort`:

```python
df['datetime'] = pd.to_datetime(df['timestamp'], format='mixed', utc=True)
df = df.sort_values('datetime')

was_completed = df['type'].shift() == 'session_complete'
time_gap      = df['datetime'].diff() > pd.Timedelta(minutes=5)
module_change = df['module'] != df['module'].shift()

df['session_index'] = (was_completed | time_gap | module_change).cumsum() + 1
```

**This is the 3-rule system:**
1. The prior row was `type='session_complete'`
2. More than 5 minutes elapsed since the prior row
3. The `module` field changed between consecutive rows

Any of the three conditions firing increments the session index. Sessions are 1-indexed (`.cumsum() + 1`).

### 3.7 Endpoint: `GET /seed`

**Auth: None — this endpoint is publicly accessible.**

**Behavior:**
1. Constructs absolute path to `lme.db` relative to the router file location
2. Deletes `lme.db` if it exists (`os.remove(db_path)`)
3. Constructs absolute path to `seed_analytics.py`
4. Runs `subprocess.run([sys.executable, script_path], cwd=..., capture_output=True, text=True)`
5. If subprocess `returncode != 0`: raises `HTTP 500` with `result.stderr`
6. On success: returns `{"status": "success", "message": "...", "output": result.stdout}`

The deletion happens **before** the subprocess, so if the subprocess fails, the database is gone and not rebuilt. `os.remove` failures are silently caught and swallowed (`except Exception as e: pass`).

## Section 4 — Discrepancies and Issues Found

### 4.1 Session Boundary Detection: 3-Rule vs Date-based

| Location | Method Used | What the architecture specifies |
|----------|------------|--------------------------------|
| `analytics.py` `/student/{id}` | **3-rule** (session_complete + 5min gap + module switch) ✅ | 3-rule system |
| `analytics.py` `/cohort` | **3-rule** ✅ | 3-rule system |
| `risk_model.py` `train_risk_model()` | **Date-based only** (`groupby('date')`) ❌ | 3-rule system |

**Impact:** High. A student who has two sessions in one day (e.g., a morning logic session and an afternoon maths session) will appear to have `session_count=1` in the ML model, but `session_count=2` in the analytics router. This means `hint_dependency_score` is also computed differently (global sum vs per-session average).

---

### 4.2 Hint Dependency Formula: Global vs Per-Session Average

| Location | Formula | Architecture Spec |
|----------|---------|-------------------|
| `risk_model.py` | `total_hints_all_sessions / total_answered_all_sessions` (global ratio) ❌ | `peak_hints / answered_count` |
| `analytics.py` `/student/{id}` | `mean(max(hints_per_session) / max(answered_per_session))` ✅ | Per-session average |
| `analytics.py` `/cohort` | `mean(max(hints_per_session) / max(answered_per_session))` ✅ | Per-session average |

**Impact:** Medium. The ML model's `hint_dependency_score` systematically underestimates hint dependency for students with many sessions because the denominator grows with each session while the numerator is bounded by `max_hints` per session.

### 4.3 ML Model: LogisticRegression, No StandardScaler

| What architecture likely specifies | What code does |
|-----------------------------------|----------------|
| SVM with StandardScaler (common in such systems) | `LogisticRegression(max_iter=500)`, no scaler ❌ |

**Impact:** Medium. Logistic Regression is sensitive to feature scale differences (e.g., `avg_SI` is in [0,1] while `session_count` can be 5+). This is mitigated by the fallback formula being used in most real deployments (< 10 training students).

### 4.7 Behavioral Trigger: `avg_idle > 120` Inconsistency

| Location | Trigger present |
|----------|----------------|
| `analytics.py` `/student/{id}` inline risk | Yes: `(float(avg_idle) > 120)` |
| `analytics.py` `/cohort` inline risk | Yes: `(s_avg_idle > 120)` |
| `risk_model.py` `predict()` | **No** — `avg_idle` is not a feature in the ML model at all ❌ |

**Impact:** Medium. A student flagged as Behavioral in the student profile / cohort view may be shown as Cognitive in the `/risk` endpoint, because the ML model does not have access to `avg_idle` at all.

---

### 4.8 `train_risk_model()` Called on Every Request

The function is called at the top of every analytics endpoint, meaning the model is **retrained from scratch on every HTTP request**. On a DB with 10 students × 400 rows this is fast, but it will not scale. There is no startup hook, no background task, and no cache invalidation strategy.

---

### 4.9 Cohort Dominant Weakness — Emotional Shortcut

As noted in Section 3.5, the cohort endpoint uses a simplified ternary:
```python
s_dominant_weakness = "behavioral" if s_is_behav else ("cognitive" if (...) else "emotional")
```

This means **emotional is the default fallback** in cohort, while **cognitive is the default fallback** in the student profile and risk model endpoints. A student with no clear signal would be labeled differently depending on which endpoint is called.

---

### 4.10 `total_mistakes` Formula — Per-Session Max, Not Per-Row Sum

| Metric | Formula used | What the name implies |
|--------|-------------|----------------------|
| `total_mistakes` | `session_maxes['total_mistakes'].sum()` — sum of the session-peak values | Sum of mistakes across all sessions |

This is correct because the seed script sets `total_mistakes` as a cumulative counter within a session (incrementing each row). Taking `max()` per session extracts the final count for that session, then summing across sessions gives the true total. **The implementation is correct** but non-obvious.
